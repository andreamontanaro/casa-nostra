## Infrastruttura

### Supabase
- Generati i tipi TypeScript dallo schema reale via MCP (`types/database.ts`)
- Tre client separati come raccomandato da `@supabase/ssr`:
  - `lib/supabase/browser.ts` — per Client Components
  - `lib/supabase/server.ts` — per Server Components e Server Actions (usa `await cookies()`)
- Profili utente inseriti direttamente in DB: **Andrea** (`higher_income: true`) e **Fede**

### Routing e autenticazione
- `proxy.ts` (ex `middleware.ts`) gestisce il refresh della sessione Supabase e i redirect:
  - utente non autenticato → `/login`
  - utente autenticato su `/login` → `/`
- Route group `app/(app)/` per le pagine protette con layout condiviso

### Variabili d'ambiente
- `.env.local` configurato localmente
- `NEXT_PUBLIC_SUPABASE_URL` e `NEXT_PUBLIC_SUPABASE_ANON_KEY` caricate su Vercel (production + development) via CLI

## Osservazioni tecniche

**Next.js 16 — breaking change middleware**
In Next.js 16 il file `middleware.ts` è deprecato. Va rinominato `proxy.ts` e la funzione esportata da `middleware` a `proxy`. Esiste un codemod ufficiale: `npx @next/codemod@canary middleware-to-proxy .`

**Tailwind 4 — niente `tailwind.config.js`**
La configurazione avviene interamente in CSS con `@theme` e `@theme inline`. I token definiti in `@theme inline` diventano classi Tailwind (es. `bg-accent`, `text-muted`). Il dark mode usa `@media (prefers-color-scheme: dark)` per default.

**React 19 — ref come prop**
`forwardRef` è deprecato in React 19. I componenti accettano `ref` direttamente come prop nel tipo dell'interfaccia, senza wrapper.

**Supabase RLS attivo**
Tutte le tabelle hanno RLS abilitato. La funzione helper `is_authorized_user()` in DB verifica che l'utente loggato sia uno dei due profili registrati. Non è necessario alcun controllo aggiuntivo lato client.

**`v_user_open_balance` è la fonte di verità per il saldo**
Il saldo non viene mai ricalcolato lato client. La vista in Postgres gestisce sia la regola 50/50 che la 60/40 e filtra solo le spese con `settlement_id IS NULL`.

**`register_settlement` è transazionale**
La RPC crea la riga in `settlements` e aggiorna tutte le spese aperte in un'unica transazione. Solleva eccezione se il saldo è zero — gestito a monte disabilitando il bottone.

## Sessione 2 — Bug fix e sicurezza DB

### Viste Supabase con SECURITY DEFINER (RLS bypassato)

**Problema.** Gli advisor di sicurezza di Supabase segnalavano `v_user_open_balance` e `v_expense_shares` come `SECURITY DEFINER`. In Postgres, le viste create da Supabase sono `SECURITY DEFINER` per default: girano con i permessi del creatore (superuser), bypassando completamente le policy RLS. Qualsiasi utente autenticato avrebbe potuto in teoria vedere i dati dell'altro.

**Soluzione.** Impostato `security_invoker = true` su entrambe le viste via migrazione:

```sql
ALTER VIEW public.v_expense_shares SET (security_invoker = true);
ALTER VIEW public.v_user_open_balance SET (security_invoker = true);
```

Con `security_invoker`, la vista gira con i permessi dell'utente chiamante: le policy RLS vengono rispettate normalmente. Disponibile su Postgres 15+ (il progetto usa Postgres 17).

Corretta contestualmente anche la funzione `set_updated_at`, che aveva `search_path` mutabile (warning di sicurezza): aggiunto `SET search_path = public` nella definizione.

---

### Falsi errori dopo eliminazione spesa e conguaglio

**Problema.** Dopo aver eliminato una spesa o registrato un conguaglio, l'app mostrava un toast di errore — ma l'operazione era già avvenuta correttamente nel database.

**Causa.** In Next.js App Router, `redirect()` funziona lanciando internamente un'eccezione speciale `NEXT_REDIRECT`. I componenti client (`EditExpenseForm`, `ConguaglioClient`) chiamavano le Server Actions dentro un `try/catch` generico che intercettava anche questa eccezione, la trattava come un errore reale, mostrava il toast e bloccava la navigazione — mentre il DB era già stato aggiornato.

**Soluzione.** Aggiunto `isRedirectError` da `next/dist/client/components/redirect-error` nei catch block: se l'eccezione è un redirect di Next.js, viene rilanciata affinché il framework la gestisca correttamente.

```ts
import { isRedirectError } from 'next/dist/client/components/redirect-error'

try {
  await deleteExpense(expense.id)
} catch (e) {
  if (isRedirectError(e)) throw e   // lascia gestire il redirect a Next.js
  toast.error('Errore durante l\'eliminazione.')
}
```

File modificati: `app/(app)/spese/[id]/EditExpenseForm.tsx`, `app/(app)/conguaglio/ConguaglioClient.tsx`.

## Sessione 4 — Divisione personalizzata per importo fisso

### Motivazione

Oltre alle regole `fifty_fifty` e `sixty_forty`, è stata aggiunta la possibilità di specificare esattamente quanto deve pagare l'altra persona. Caso d'uso tipico: ristorante, dove ognuno consuma importi diversi. Chi inserisce la spesa indica la quota dell'altra persona; il resto va automaticamente a chi ha pagato.

### DB — Migrazione

**Nuovo valore enum:**
```sql
ALTER TYPE split_rule ADD VALUE 'custom';
```

**Nuova colonna `custom_other_share`:**
```sql
ALTER TABLE public.expenses
  ADD COLUMN custom_other_share numeric(10,2)
  CONSTRAINT expenses_custom_other_share_positive
    CHECK (custom_other_share IS NULL OR custom_other_share > 0);
```
Nullable per le spese esistenti. Obbligatoria — via constraint — se e solo se `split_rule = 'custom'`.

**Constraint di coerenza:**
```sql
ALTER TABLE public.expenses
  ADD CONSTRAINT expenses_custom_share_consistency CHECK (
    (split_rule = 'custom' AND custom_other_share IS NOT NULL) OR
    (split_rule <> 'custom' AND custom_other_share IS NULL)
  );
```

**Vista `v_expense_shares` aggiornata** per gestire la nuova regola: chi non ha pagato riceve `custom_other_share` come quota; chi ha pagato riceve `amount - custom_other_share`. Il calcolo in `v_user_open_balance` e la RPC `register_settlement` rimangono invariati — la logica è tutta nella vista.

### TypeScript

`types/database.ts` aggiornato manualmente (CLI Supabase richiede access token non disponibile):
- Aggiunta `custom_other_share: number | null` in `Row`, `Insert`, `Update` di `expenses`
- Aggiunto `'custom'` nell'enum `split_rule` e nel `Constants` array

### Frontend

- **`lib/fmt.ts`** — aggiunto `custom: 'Personalizzato'` in `SPLIT_LABELS`
- **`ExpenseForm.tsx`** / **`EditExpenseForm.tsx`** — terzo chip "Personalizzato" nella sezione Divisione; quando selezionato, appare il campo "Quota di [nome] (€)" con anteprima in tempo reale della quota residua ("La tua quota: €X")
- **`app/actions/expenses.ts`** — validazione (`> 0` e `< amount`) e persistenza di `custom_other_share`
- **`spese/[id]/page.tsx`** — nella stat "Divisione" mostra `Personalizzato (€X)` quando `split_rule === 'custom'`
